# ADR-0002: Adopt Microservices Architecture

## Status
Accepted

## Context
The Sentio system has distinct components with vastly different requirements:
1. **IoT/Embedded:** Low bandwidth, C++/Python logic.
2. **AI Inference:** CPU/GPU intensive, Python ecosystem (PyTorch).
3. **Backend Business Logic:** I/O bound, transactional, Java ecosystem.
4. **Frontend:** Client-side rendering.

A monolithic approach would force a single technology stack and make scaling specific parts (like the AI inference) difficult.

## Decision
We will adopt a **Microservices Architecture**. The system will be split into loosely coupled services: `sentio-backend` (Core), `sentio-ai` (Inference), `sentio-web` (UI), and `sentio-embedded` (Edge).

## Consequences
**Positive:**
* **Polyglot programming:** We can use the best language for the job (Python for AI, Java for Backend).
* **Independent Scalability:** We can scale AI services separately from the CRUD backend.
* **Isolation:** A failure in the AI service does not crash the core device management.

**Negative:**
* Increased operational complexity (deployment, networking).
* Requires strict API contracts between services.
