# ADR-0012: Layered Architecture Pattern

## Status
Accepted

## Context
Within the `sentio-backend`, mixing database code, business logic, and HTTP handling in single classes makes code hard to test and maintain. 
We need a clear separation of concerns.

## Decision
We will use a standard **Layered Architecture**:
1. **Controller Layer:** Handles HTTP requests/responses.
2. **Service Layer:** Contains business logic.
3. **Repository Layer:** Handles data access (JPA).
4. **Domain Layer:** Defines entities.

## Consequences
**Positive:**
* **Testability:** We can unit test Services by mocking Repositories.
* **Maintainability:** Changing a database query doesn't affect the controller.
* **Standardization:** Familiar pattern for Spring Boot developers.

**Negative:**
* Write slightly more boilerplate code (DTOs, entity mappings).
