# ADR-0010: RESTful API Design Pattern

## Status
Accepted

## Context
We need a standard communication interface between the Frontend, Backend, and AI services. It must be language-agnostic, cacheable, and easy to document.

## Decision
We will use the **REST (Representational State Transfer)** architectural style for synchronous service communication, defined via **OpenAPI 3.0** specifications.

## Consequences
**Positive:**
* Universal support across all platforms.
* Statelessness improves scalability.
* OpenAPI allows for contract testing and client code generation.

**Negative:**
* Can lead to over-fetching or under-fetching of data (compared to GraphQL), though acceptable for this scope.
