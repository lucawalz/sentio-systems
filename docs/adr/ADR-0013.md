# ADR-0013: GitHub Actions CI/CD Pipeline Design

## Status
Accepted

## Context
The Sentio Systems monorepo spans multiple technology stacks (Java/Spring Boot, TypeScript/React, Python/FastAPI, Python/embedded). Every change needs to be automatically validated, built, and optionally deployed without manual intervention. We needed a CI/CD solution that:

- Integrates natively with our GitHub-hosted repository
- Supports our polyglot monorepo without running irrelevant jobs on unrelated changes
- Enforces our test pyramid (unit → integration) with clear fast-fail gates
- Publishes container images for all services to a registry
- Can be extended with dependency automation

## Decision

### 1. GitHub Actions (not GitLab CI or Jenkins)
We use **GitHub Actions** because the repository is already hosted on GitHub. This eliminates the need for an external CI server and provides native integration with pull requests, security scanning (SARIF upload), package registry (GHCR), and Dependabot.

### 2. Conditional Job Execution via Path Filtering
The monorepo contains four independently deployable components. Running all jobs on every commit would be wasteful and slow. We use `dorny/paths-filter@v3` to detect which parts of the repository changed:

```
sentio-backend/**  → backend-test, backend-build, backend-docker
sentio-web/**      → frontend-test, frontend-build, frontend-docker
sentio-ai/**       → ai-services-test, ai-services-docker
sentio-embedded/** → embedded-test
```

**Exception:** On `main`, `develop`, and `release/**` branches, all jobs run unconditionally regardless of changed paths. This ensures that integration-branch commits always produce a full quality report and deployable artifacts.

**Rationale:** A developer fixing a typo in a Python file should not wait for a 10-minute Java build. Path filtering reduces average CI time by ~70% for single-component changes while still enforcing full coverage on integration branches.

### 3. Separate Unit and Integration Test Phases
Backend tests are split into two sequential Maven steps:

1. **Unit tests** (`*Test` pattern, timeout 5 min): Pure JUnit/Mockito tests with no external dependencies. These run first and act as a fast-fail gate — if business logic is broken, the developer gets feedback within minutes without waiting for container startup.

2. **Integration tests** (`*IT,SentioApplicationTests` pattern, timeout 10 min): Full `@SpringBootTest` tests using Testcontainers to spin up a real PostgreSQL instance. These verify end-to-end behaviour through the complete service and repository layers. `SentioApplicationTests` (context load smoke test) is included here because it also requires Testcontainers.

3. **Coverage verification** (`mvnw jacoco:report` + threshold check): Runs after both test phases have completed. Enforces a minimum of 50% instruction coverage and 35% branch coverage on the combined report.

**Rationale:** This ordering gives the fastest possible developer feedback loop: unit test failures are surfaced before Docker images are pulled for Testcontainers, reducing mean time to red from ~8 minutes to ~2 minutes.

### 4. Multi-Architecture Docker Builds (amd64 + arm64)
All Docker images are built for both `linux/amd64` and `linux/arm64` platforms. The Sentio edge devices are Raspberry Pi (ARM64) while CI runners and cloud deployments use AMD64. Building both architectures in CI eliminates platform-specific runtime failures discovered late in the deployment process.

### 5. Automatic GitHub Releases on `main`
Every push to `main` triggers a GitHub Release with an auto-calculated semantic version. The version is derived from the latest git tag:
- `main` → minor bump (new feature releases)
- `develop` → patch bump with `-dev.<timestamp>` pre-release suffix
- `release/**` → patch bump with `-rc.<timestamp>` pre-release suffix

Timestamp-based pre-release identifiers (instead of commit hashes) are used because semver ordering is chronological — `dev.1703001234 < dev.1703001300` is guaranteed, whereas commit hashes have no chronological order.

### 6. Dependabot for Dependency Automation
Dependabot is configured for all five package ecosystems (GitHub Actions, Maven, npm, pip × 3 AI services, Docker × 5 images) with weekly update schedules. Major version updates are suppressed for Spring Boot (breaking changes) and all npm packages (frontend stability). This keeps transitive dependencies patched against CVEs without requiring manual intervention.

### 7. Security Scanning with Trivy
Trivy filesystem scans run on every commit independent of path filters. Image scans run after each Docker build. Results are uploaded as SARIF to GitHub Security tab for visibility. Scans are configured with `ignore-unfixed: true` to suppress noise from vulnerabilities with no available fix.

## Consequences

**Positive:**
- Average CI time for a single-component change is ~3–5 minutes instead of ~15 minutes for a full monorepo build.
- Full test suite (unit + integration) always runs on `main` and `develop`, ensuring no regressions reach shared branches.
- GHCR images are automatically tagged with semantic versions, short SHAs, and branch names, enabling flexible deployment strategies.
- Security vulnerabilities in dependencies are surfaced automatically via Dependabot PRs.

**Negative:**
- Path filtering requires the `dorny/paths-filter` action. If that action is deprecated, the conditional logic must be rewritten.
- Multi-architecture builds roughly double Docker build time (~20 min instead of ~10 min for AI service images with large Python dependencies).
- Two separate Maven test commands mean JaCoCo `.exec` files are not accumulated across phases. The final coverage report reflects IT test coverage only; the manual `jacoco:report` step after both phases uses the last run's data. Mitigation: IT tests use `@SpringBootTest` which exercises the full service stack, so reported coverage is representative.
