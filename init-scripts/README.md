# Keycloak Automatic Configuration

This directory contains initialization scripts for automatic setup of PostgreSQL and Keycloak.

## Files

### `init-db.sh`
- Automatically creates the `keycloak` database in PostgreSQL
- Runs on first startup of the postgres container
- The `sentio` database is auto-created by PostgreSQL via `POSTGRES_USER` environment variable

### `sentio-realm.json.template`
- **Template** for Keycloak realm configuration (committed to git)
- Contains placeholder `KEYCLOAK_CLIENT_SECRET_PLACEHOLDER` for the client secret
- The `setup-env.sh` script generates the actual `sentio-realm.json` from this template

### `sentio-realm.json` (Generated, not in git)
- Actual Keycloak realm configuration with real client secret
- Generated by `setup-env.sh` from the template
- **Gitignored** to keep secrets out of version control
- Contains pre-configured:
  - **Realm:** `sentio`
  - **Client:** `sentio-backend` with generated client secret
  - **Roles:** `user` and `admin`
  - Security settings and session configurations

## How It Works

### PostgreSQL Initialization
The postgres container uses the `/docker-entrypoint-initdb.d` directory to run initialization scripts on first startup:
```yaml
volumes:
  - ./init-scripts:/docker-entrypoint-initdb.d
```

### Keycloak Realm Import
Keycloak automatically imports realm configurations on startup using the `--import-realm` flag:
```yaml
command: start-dev --import-realm
volumes:
  - ./init-scripts/sentio-realm.json:/opt/keycloak/data/import/sentio-realm.json
```

## Fresh Deployment

To deploy with a fresh setup:

```bash
# 1. Generate environment configuration and realm JSON
./setup-env.sh

# 2. Stop and remove all containers and volumes (if any exist)
docker-compose down -v

# 3. Start everything (realm will be imported automatically)
docker-compose up -d
```

The `setup-env.sh` script will:
1. Generate secure random passwords
2. Create `.env` file with these credentials
3. Generate `sentio-realm.json` from template with the client secret

The `sentio` realm and `sentio-backend` client will be automatically created when Keycloak starts.

## Client Secret Management

The client secret workflow:

1. **`setup-env.sh`** generates a random client secret
2. Stores it in **`.env`** as `KEYCLOAK_ADMIN_CLIENT_SECRET`
3. Generates **`sentio-realm.json`** from **`sentio-realm.json.template`** with the secret
4. Keycloak imports the realm with this secret on startup
5. Backend reads the secret from `.env` environment variable

**These match automatically!** Both Keycloak and the backend use the same secret from the initial generation.

### If You Need to Regenerate the Secret

```bash
# 1. Run setup script again (it will ask to overwrite)
./setup-env.sh

# 2. Restart services to apply new configuration
docker-compose down -v
docker-compose up -d
```

**Important:** The template file (`sentio-realm.json.template`) is in git with a placeholder. The actual file (`sentio-realm.json`) with the real secret is gitignored.

## Security Notes

- The realm configuration file contains the client secret
- **DO NOT commit sensitive secrets in production environments**
- For production, use different secrets per environment
- Consider using secret management tools (Vault, AWS Secrets Manager, etc.)
- The current setup is designed for development convenience

## Production Considerations

For production deployments:

1. **Use environment-specific secrets:**
   - Create `sentio-realm-prod.json` with production secrets
   - Don't commit production secrets to git
   - Use secret management services

2. **Persistent Keycloak data:**
   - Add a volume for Keycloak data to preserve users/settings
   - The import only happens if the realm doesn't exist

3. **SSL/TLS:**
   - Set `"sslRequired": "external"` or `"all"` in realm config
   - Configure proper certificates

4. **Disable dev mode:**
   - Use `command: start` instead of `start-dev`
   - Configure production-ready settings
